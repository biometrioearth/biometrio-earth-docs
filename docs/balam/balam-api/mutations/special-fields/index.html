Special fields

When updating or creating you may encounter some errors in specific fields in certain models. That's because some fields have custom scalars due to not being defined by default in the graphql schema or because it was easier to defined a custom scalar for the to have a better syntax when writing the query.

JSON scalar​

We created a custom JSON Scalar because Graphene doesn't provide a definition for this type of field. For this custom scalar we let the syntax be just the same as posible as a JSON object, the only difference is that the keys the JSON object doesn't go double quoted.

mutation {
  createSite(
    identifier: "site 1",
    project: "56yda30e-96eb-65kl-568k-1d0421ad360c",
    metadata: {
      vegetation: "sparse",
      treesCount: 10
    }
  ) {
    id
  }
}

you can also add dates but values must respect timestamp syntax: yyyy-MM-ddTHH:mm:ss so filters can work.

info

When working with this type of scalar you may encounter some errors due to JSON parsing, but this depends on the programming language that you are using to make the requests. Check the bottom section to see some examples on how you can make requests with this type of scalar.

Date scalars​

The DateTime scalar type represents a DateTime value as specified by iso8601, which sets the format as yyyy‐MM‐ddTHH:mm:ssZ, same with the Date scalar, the format would be yyyy‐MM‐dd. For now, balam only uses the DateTime scalar, an example of a mutation with this scalar is the following

mutation {
  createSamplingPoint(
    identifier: "sampling point 1",
    project: "56yda30e-96eb-65kl-568k-1d0421ad360c",
    dateDeployment: "2023-06-19T14:05:34Z"
  ) {
    id
  }
}

GIS scalars​

At the moment we only use two GIS scalars in the Balam models, and those are Polygon and Point. Both share almost the same syntax, but differ in the coordinates property of the geojson. The generic syntax of the geojson for the GIS scalars is the next one:

{
    "type": "Point", // or Polygon
    "coordinates": [
        ...
    ]
}

to use it in a query you can write:

mutation {
    createSite(
    identifier: "site 1"
    geometry: {
      type: "Polygon",
      coordinates: [
        ...
      ]
    }
  ) {
    id
  }
}

the coordinates in the Point and Polygon change because for points we only need the latitude and longitude, and for polygons we need more information.

Point​

This GIS scalar is the most simple one, because we only require the latitude and longitude. Ypu can see an example of the syntax for this scalar in the next mutation:

mutation {
    createSamplingPoint(
    identifier: "point 1"
    location: {
      type: "Point",
      coordinates: [
        100.799722222222,
        1.03805555555555
      ]
    }
  ) {
    id
  }
}

keep in mind that the first element of the coordinates array is the longitude and the second one is the latitude.

Polygon​

A Polygon is a GIS object that stores a series of x and y coordinate pairs that enclose an area. So using this knowledge and the examples before a Polygon scalar is written like this:

mutation {
    createSite(
    identifier: "site 1"
    location: {
      type: "Polygon",
      coordinates: [
        [
            [
              -102.43244757476604,
              26.19399923194422
            ],
            [
              -102.43244757476604,
              24.29590720167262
            ],
            [
              -100.09089358417015,
              24.29590720167262
            ],
            [
              -100.09089358417015,
              26.19399923194422
            ],
            [
              -102.43244757476604,
              26.19399923194422
            ]
          ]
      ]
    }
  ) {
    id
  }
}
caution

Do not forget that the coordinates of the Polygon are inside of a nested array and the first and last coordinate is the same, this to close the polygon. So the array should look like:

coordinates: [
    [
        [coordinates array point 1],
        ...
        [coordinates array point 1]
    ]
]

and as with the Point scalar, the first element of each coordinate pair represents the longitude and the second one the latitude.

Mutation Examples​

For the GIS and JSON scalars sometimes the parsing can be a little annoying, so here we present some examples to know how to work with this type of scalars.

JSON​

When we work with JSON scalar in the graphiql interface we can write the JSON just as similar as working in javascript:

{
  propertyOne: "somevalue",
  propertyTwo: 123,
  propertyThree: "2023-07-19T14:34:23Z",
  propertyFour: false
  ...
}

just remember to not quote the properties.

This rule of not quoting the properties must be followed when making the mutation request with js or python. But sometimes when parsing the request, the language and method that we use to make the parse adds the quotation to the property automatically, so to avoid that, you can add the JSON scalar like this:

  • JavaScript
  • Python
let data = JSON.stringify({
  "query": `
    mutation {
      createSamplingPoint(
        identifier: "sp 1",
        project: "56yda30e-96eb-65kl-568k-1d0421ad360c",
        metadata: {
          exampleOne: "somevalue",
          exampleTwo: 123
        }
      ) {
        id
      }
    }
  `
})

fetch("<BALAM API ENDPOINT>", {
  headers: {
    "Content-Type": "application/json",
    "Authorization": "JWT <token>"
  },
  data: data
}).then(response => { return response.json() })
  .then(data => { console.log(data) })
data = """
  mutation {
    createSamplingPoint(
      identifier: "sp 1",
      project: "56yda30e-96eb-65kl-568k-1d0421ad360c",
      metadata: {
        exampleOne: "somevalue",
        exampleTwo: 123
      }
    ) {
      id
    }
  }
"""
response = requests.post("<BALAM API ENDPOINT>",
    headers: {
      "Authorization": "JWT <token>"
    },
    json={
        "query": data
    })

print(response.json())

But what if you want to write the mutation using a proper syntax for graphql mutations?, then you'll need to create a dictionary or an object depending on your programming language:

  • JavaScript
  • Python
// Define the GraphQL mutation
const mutation = `
  mutation CreateSamplingPoint(
    $identifier: String,
    $project: ID!,
    $metadata: JSONObject
  ) {
    createSamplingPoint(
      identifier: $identifier,
      project: $project,
      metadata: $metadata
    ) {
      id
    }
  }
`

// Define the variables for the mutation
const variables = {
  identifier: "sp 1",
  project: "56yda30e-96eb-65kl-568k-1d0421ad360c",
  metadata: {
    exampleOne: "somevalue",
    exampleTwo: 123
  }
};

// Create the request body
let data = JSON.stringify({
  "query": mutation,
  "variables": variables
})
# Define the GraphQL mutation
mutation = '''
  mutation CreateSamplingPoint(
    $identifier: String,
    $project: ID!,
    $metadata: JSONObject
  ) {
    createSamplingPoint(
      identifier: $identifier,
      project: $project,
      metadata: $metadata
    ) {
      id
    }
  }
'''

# Define the variables for the mutation
variables = {
  'identifier': 'sp 1',
  'project': '56yda30e-96eb-65kl-568k-1d0421ad360c',
  'metadata': {
    'exampleOne': 'somevalue',
    'exampleTwo': 123
  }
}

# Create the request payload
payload = {
    'query': mutation,
    'variables': variables
}

The problem here is that for JavaScript when we use the method JSON.stringify(), the output of that would be

JSON.stringify()
'{"query":"\\n  mutation CreateSamplingPoint(\\n    $identifier: String,\\n    $project: ID!,\\n    $metadata: JSONObject\\n  ) {\\n    createSamplingPoint(\\n      identifier: $identifier,\\n      project: $project,\\n      metadata: $metadata\\n    ) {\\n      id\\n    }\\n  }\\n","variables":{"identifier":"sp 1","project":"56yda30e-96eb-65kl-568k-1d0421ad360c","metadata":{"exampleOne":"somevalue","exampleTwo":123}}}'

and taking a closer look: 

"metadata":{"exampleOne":"somevalue","exampleTwo":123}

which has the keys quoted, and would return an error. To avoid that you can create a function that stringifies the object without adding the quotes in the keys and use that function when defining the variables for the mutation:

JavaScript
function stringifyWithoutQuotes(obj) {
  const keyValuePairs = [];

  for (const key in obj) {
    if (obj.hasOwnProperty(key)) {
      const value = obj[key];
      const serializedValue = JSON.stringify(value);
      keyValuePairs.push(`${key}:${serializedValue}`);
    }
  }

  return `{${keyValuePairs.join(",")}}`;
}

// Define the variables for the mutation
const variables = {
  identifier: "sp 1",
  project: "56yda30e-96eb-65kl-568k-1d0421ad360c",
  metadata: stringifyWithoutQuotes({
    exampleOne: "somevalue",
    exampleTwo: 123
  })
};

...

whith this you will avoid the quoted keys and your mutation will be succesful.

tip

If you use Apollo client of some other GraphQL client, you probably don't need any of this, because with this clients you can write your mutations just the same as you would in the GraphiQL.

GIS​

For GIS scalars the sytnax is the same as with the JSON ones, so you'll need to write your Polygons or Points like a geojson with unquoted keys.

You can also write your GIS scalars with WKT syntax, if you do that, you just need to write your value like a string:

mutation {
    createSamplingPoint(
    identifier: "point 1"
    location: "POINT (100.799722222222 1.03805555555555)"
  ) {
    id
  }
}
